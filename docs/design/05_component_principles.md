# Component Principles

## 目的

コンポーネントは、単に UI の部品ではなく、
`この会社らしい見え方` を積み上げる単位として設計する。

## Navigation

### 方針

- 固定表示でも軽く見えないこと
- コーポレート感より、スタジオ感を出すこと

### ルール

- ロゴ周辺はコンパクトに強く
- ナビ項目は多すぎない
- CTA を右端に置く場合は1つに絞る

## Hero Preview Card

### 役割

- 会社の専門領域を視覚的に伝える
- 単なる飾りで終わらせない

### 中に入れるべきもの

- 何を作るか
- 何を重視しているか
- データやプロダクトの断片

### ルール

- 中央に主役感を持たせる
- テキストは短く
- 装飾だけのカードにしない

## Strength Card

### 役割

- 他社との差分を端的に伝える

### ルール

- タイトルだけで意味が分かる
- 本文は1テーマだけを話す
- 3枚なら、3枚とも重複しない

## Process Card

### 役割

- 進め方の不安をなくす

### ルール

- ステップ番号を明確にする
- 各カードに役割の違いを持たせる
- UI の遊びより、理解しやすさを優先する

## Achievement Card

### 役割

- 実績の説得力を出す

### ルール

- ラベル、タイトル、説明の3層で整理する
- タイトルに具体名を入れる
- 抽象的な自慢で終わらせない

## Team Card

### 役割

- 人を見せて信頼感を出す
- 何を担当できる人かを短時間で理解させる

### ルール

- 顔、役割、強みを明確に分ける
- 実績は箇条書きで見せる
- 自己紹介文は長くしすぎない

## CTA

### 方針

- 何をすればよいか迷わせない

### 主CTA

- 相談する
- 実績を見る

### 避けるべきCTA

- 詳しくはこちら
- お問い合わせ
- Learn more

抽象的で行動が見えないためです。

## フォームや問い合わせ導線

### 必要なこと

- 曖昧な相談でもよいと書く
- 相談対象を具体的に書く
- 不要に項目を増やさない

### 望ましい補助文

- 新規事業の相談
- AI 活用の試作
- データ活用の整理
- プロダクトの初期設計
